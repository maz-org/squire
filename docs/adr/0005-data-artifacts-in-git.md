## Context

Pre-migration, two data artifacts were committed to the repo:

- **`data/index.json`** — serialized Xenova rulebook embeddings, ~11MB,
  regenerated by `npm run index`. A binary-ish blob that churned on every
  reindex and bloated every clone and every PR.
- **`data/extracted/*.json`** — structured card data produced by
  `import-*.ts` scripts from upstream GHS (Gloomhaven Secretariat).
  Text-diffable, refreshed by a weekly GitHub Actions job.

ADR-0004 established that Postgres is the runtime source of truth, so neither
file is read at runtime after the storage migration. The question is which of
them should continue to live in git as a source artifact.

## Decision

**Delete `data/index.json` from git. Keep `data/extracted/*.json` committed.**
The two artifacts serve different purposes and deserve different fates.

`data/index.json` is a derived binary blob — regenerable in ~60s from
`data/pdfs/*.pdf` — with no diff value to a human reviewer. It goes.

`data/extracted/*.json` is the change log for upstream GHS card data. Weekly
CI refreshes it and opens a PR. That PR is the mechanism by which a human
(Brian) audits what GHS changed upstream before it hits production. Deleting
it would delete that audit trail.

## Options considered

- **Option A** (chosen): Delete `index.json`, keep `extracted/*.json`. Removes
  11MB of churn; preserves the human-reviewable weekly GHS diff. New-dev
  bootstrap is `docker-compose up && npm run index && npm run seed:cards`
  (~2 min).
- **Option B**: Keep both. Rejected: `index.json` churn is the main pain point
  and it has no review value.
- **Option C**: Delete both, regenerate `extracted/*.json` on-demand. Rejected:
  loses the weekly human-reviewable diff of upstream changes, which is the
  primary defense against silent upstream data regressions.
- **Option D**: Keep `index.json` out of git via git-lfs. Rejected: adds
  infrastructure for a file that has zero review value.

## Consequences

- **Easier:** cleaner clones and PRs (no 11MB blob); upstream GHS changes stay
  visible in weekly diffs; Dependabot-style "review the changes before they
  land" works the same way for data as it does for deps.
- **Harder:** new-dev bootstrap grows a reindex step (~60s). Acceptable — the
  total bootstrap target is under 3 minutes.
- **Re-evaluate if:** the reindex step becomes a routine pain point, or if
  `extracted/*.json` grows large enough that diff review stops being viable.

## Advice

Decided during `plan-eng-review` on 2026-04-07 (Decisions 7 and 8 in
`docs/plans/storage-migration-review-checkpoint.md`). `docs/DEVELOPMENT.md` and
`docs/CONTRIBUTING.md` were updated to describe the new bootstrap;
`.github/workflows/refresh-data.yml` was updated to stop touching
`data/index.json`.
