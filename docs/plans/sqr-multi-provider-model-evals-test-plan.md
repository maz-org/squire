## Purpose

This plan defines the required test coverage for the eval-only multi-provider
model comparison project. The implementation must keep production `/api/ask`,
REST, MCP, and web behavior unchanged until the eval report explicitly justifies
production work.

Langfuse is the authoritative trace and debugging system for this project.
Local files may exist as convenience exports or summaries, but replay, diffing,
and report tooling should read from Langfuse run or trace data.

## Required Test Files

- `test/eval-cli.test.ts`
  - Parses provider/model, run label, timeout, max output, reasoning effort, and
    tool loop limit.
  - Applies CLI-over-environment precedence.
  - Rejects unsupported provider/model combinations with useful errors.
  - Keeps existing `--tool-surface` behavior until it is intentionally retired.

- `test/eval-openai-schema.test.ts`
  - Renders every Squire agent tool as an OpenAI strict function schema.
  - Rejects future tools that cannot be represented safely.
  - Removes unsupported `default` fields.
  - Adds `additionalProperties: false` to every object schema.
  - Converts optional inputs such as `topK`, `limit`, `filter`, `relation`, and
    `linkType` into required nullable fields.
  - Normalizes `null` back to existing default or `undefined` behavior before
    calling `executeToolCall`.

- `test/eval-trace.test.ts`
  - Builds Langfuse-bound trace payloads with provider, model, run label,
    dataset case, prompt version, tool schema version, stop/status reason,
    token usage, cost estimate, timing, retries, tool calls/results, errors,
    and final answer.
  - Redacts API keys, bearer tokens, cookies, session-like fields, and future
    user/campaign state fields before trace data is recorded.
  - Verifies Langfuse metadata is filterable by provider, model, dataset case,
    and run label.
  - Preserves provider-native transcript items only in eval trace data, not app
    conversation history.

- `test/eval-anthropic-runner.test.ts`
  - Runs the existing Claude loop with an eval-only model override.
  - Proves Sonnet 4.6 and Opus 4.7 use the same tool loop and differ only by
    model config.
  - Verifies production `AskOptions` does not expose provider/model selection.
  - Classifies Anthropic API access, timeout, tool execution, and answer-quality
    failures.

- `test/eval-openai-runner.test.ts`
  - Uses a manual stateless Responses loop.
  - Sends tool outputs back with the provider-native response items required for
    tool continuation.
  - Does not depend on `previous_response_id` or hosted response retention.
  - Executes Squire tools through the shared strict schema renderer and
    `executeToolCall`.
  - Classifies model access, API status, schema, tool execution, timeout, and
    answer-quality failures.

- `test/eval-matrix.test.ts`
  - Runs one selected case across every configured provider/model.
  - Supports `--id`, `--category`, and full dataset modes without duplicated
    runner logic.
  - Preserves successful model results when another model fails.
  - Emits summaries with score, pass/fail, latency, token usage, estimated cost,
    tool-call count, retry count, and Langfuse trace/run links.

- `test/eval-replay.test.ts`
  - Replays one failed case from Langfuse run/trace data.
  - Prints a readable transcript with prompts, stop/status reasons, tool calls,
    sanitized tool results, retries, final answer, judge output, and failure
    classification.
  - Diffs Claude and OpenAI traces for the same case and highlights divergent
    tool choice, missing retrieval, loop cutoff, API/schema failure, or final
    answer mismatch.

- `test/eval-cost-harness.test.ts`
  - Compares two named runs and reports accuracy, latency, token, cost, retry,
    timeout, loop iteration, and tool-call deltas.
  - Records model-specific knobs in trace and summary metadata.
  - Makes incompatible run comparisons fail with a useful explanation when
    prompt or tool schema versions differ.

## Live API Checks

Unit and integration tests must mock Anthropic, OpenAI, and Langfuse clients.
Live API calls are limited to `SQR-123` verification and any explicitly approved
manual/full-dataset eval runs. Live checks must record exact model IDs,
capabilities, rate-limit blockers, and fallback decisions in the capability
matrix.

## Required Failure Modes

- Missing or unavailable model ID.
- Unsupported OpenAI strict schema.
- Tool execution throws.
- Tool result cannot be serialized safely.
- Model returns multiple tool calls in one turn.
- Model reaches loop limit.
- Provider API timeout.
- Judge returns malformed JSON.
- Langfuse write fails.
- Partial matrix failure with other model results still usable.
- Trace payload contains a secret before redaction.
- Attempted replay references a missing Langfuse run or trace.

## Per-Issue Test Gates

- `SQR-123`: live verification artifact exists, and any missing capability has a
  documented fallback or kill criterion.
- `SQR-124`: CLI/config parsing tests pass, and `eval/run.ts` is reduced to a
  thin entrypoint before provider runner work starts.
- `SQR-125`: checked-in trace contract maps every required report/debugging
  field to Langfuse metadata, generation, span, score, or optional export.
- `SQR-126`: strict schema renderer tests cover every Squire tool.
- `SQR-127`: trace shape and redaction tests pass before provider runners depend
  on trace data.
- `SQR-128`: Anthropic runner tests prove eval-only model override without
  production `AskOptions` changes.
- `SQR-129`: OpenAI runner tests prove stateless tool continuation and failure
  classification.
- `SQR-131`: matrix tests prove partial failures are visible and non-fatal.
- `SQR-132`: replay/diff tests prove Langfuse-backed diagnosis for one failed
  case.
- `SQR-133`: cost harness tests prove comparable named runs and incompatible-run
  rejection.
- `SQR-134`: report generation links representative Langfuse traces for wins,
  losses, and confusing cases.
- `SQR-135`: no tests required unless `SQR-134` recommends production
  multi-provider support, in which case the new production plan must define its
  own test matrix.
