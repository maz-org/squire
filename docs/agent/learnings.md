# Curated Learnings

This file is the checked-in synthesis layer for durable learnings promoted
out of `~/.gstack/projects/maz-org-squire/learnings.jsonl`.

It is intentionally curated, not a raw dump. Put non-obvious, repeated,
high-signal lessons here when they should survive tool-local runtime state.

If a learning turns into a real architecture decision, write an ADR instead
of treating this file as the permanent decision record.

## Pitfalls

- **auth-pages-must-use-design-system** (pitfall): Any page the user can see, including error pages and auth flow pages, must use the project layout shell and design system. Inline HTML with system fonts and hardcoded colors creates a jarring visual break. The eng review should flag new HTML-rendering code that does not go through the layout shell. Files: `src/server.ts`, `src/web-ui/auth-error-page.ts`. Source: `observed`.
- **secure-cookie-breaks-localhost** (pitfall): Hono Secure cookie attribute prevents cookies from sticking over plain HTTP on localhost. Must conditionally set Secure=true only in production or when URL scheme is HTTPS. Standard web dev pitfall but easy to miss in plan reviews that specify Secure unconditionally. Files: `src/auth/session-middleware.ts`. Source: `cross-model`.
- **adr-number-collision** (pitfall): Before picking a new ADR number, always 'ls docs/adr/' first. SQR-71 created 'docs/adr/0009-on-demand-asset-pipeline.md' without checking — main already had '0009-google-oauth-with-hardcoded-allowlist.md' and '0010-current-turn-ledger.md'. Two ADRs with the same number would have been a confusing tangle of broken links forever. Caught only by /document-release's doc discoverability sweep; /review, /qa, and the eng review all missed it because none of them listed the ADR directory. Renamed to 0011 in commit 499b67c but all the references (frontmatter id, ADR 0008 superseded_by, 11 code comments, 4 doc files, CI comment, Linear issue) had to be updated in one pass. Files: `docs/adr/`. Source: `observed`.
- **prod-imports-in-deps-not-devdeps** (pitfall): If the production server imports a package at runtime, it belongs in 'dependencies', not 'devDependencies' — regardless of whether the current deploy uses 'npm ci --omit=dev'. SQR-71 put @tailwindcss/node, @tailwindcss/oxide, and tailwindcss in devDependencies because the old @tailwindcss/cli lived there, but after switching to in-process compile at request time they became runtime deps. A future Dockerfile adding the standard '--omit=dev' optimization would have crashed the server on startup with MODULE_NOT_FOUND. Rule: package placement follows "is this imported at runtime?" not "does current deploy currently strip devDeps?" CodeRabbit caught this on PR #205. Files: `package.json`, `src/web-ui/assets.ts`. Source: `observed`.
- **hono-regex-param-suffix** (pitfall): Hono 4.12 RegExpRouter does not support the shape ':param{regex}.literal' (regex constraint followed by a literal suffix in the same path segment). Single-segment variants like '/app.:hash{[a-f0-9]+}.css' silently 404 on matching inputs. Multi-segment variants like '/assets/:hash{[a-f0-9]+}.css' throw an uncaught 'TypeError: undefined is not iterable' from 'buildMatcherFromPreprocessedRoutes' at first request. The supported pattern is a full-filename regex constraint: '/:file{app\.[a-f0-9]+\.css}'. Hono docs show this at hono.dev/docs/api/routing#parameters with '/:filename{.+\.png}'. Burned ~30 min on SQR-71 debugging phantom 404s before finding the workaround. Files: `src/server.ts`, `docs/adr/0011-on-demand-asset-pipeline.md`. Source: `observed`.
- **gstack-learnings-search-limit-misleading** (pitfall): gstack-learnings-search --limit N does NOT mean show top N after dedup. It scans only the first N raw lines from learnings.jsonl, then dedupes that subset, then prints. So --limit 5 returns at most 5 entries even if the file has 20 unique keys, and silently undercounts. To get a real total, always use --limit 100 (or higher) when reporting counts. Better signal: jq length of the raw file (wc -l ~/.gstack/projects/$SLUG/learnings.jsonl) for raw entries, or pipe through jq -r .key | sort -u | wc -l for unique. Burned an entire /learn report on this — told the user we had 7 learnings when the real number was 19. Source: `observed`.

## Patterns

- **rails-propshaft-dev-prod-split** (pattern): For on-demand compiled assets (CSS, JS islands), Squire uses Rails Propshaft semantics: dev serves bare paths ('/app.css', '/squire.js') with 'Cache-Control: no-cache' for clean devtools and immediate edit-refresh; prod serves content-hashed paths ('/app.<hash>.css', '/squire.<hash>.js') with 'Cache-Control: public, max-age=31536000, immutable' so Cloudflare and browsers cache forever and invalidation is automatic on content change. Hash is first 10 hex chars of sha256(content) — matches Vite/Webpack defaults. Routes for both shapes are registered unconditionally; handlers branch on 'NODE_ENV' at request time so tests can stub env per case without re-importing the server module. Promise memoization on the compile path prevents cold-start races (two concurrent first-requests share one compile via in-flight Promise). Key-drift guard: after awaiting an in-flight promise, re-check the cache key matches the current caller's key; fall through to a fresh compile on drift. See ADR 0011 and src/web-ui/assets.ts. Files: `src/web-ui/assets.ts`, `src/server.ts`, `docs/adr/0011-on-demand-asset-pipeline.md`. Source: `observed`.
- **coderabbit-incremental-rounds** (pattern): CodeRabbit reviews each commit incrementally — every push triggers a new review pass. Expect 3-5 rounds on a non-trivial PR even when each round only finds 1-3 issues. After fixing and pushing, wait for the NEXT review pass on the new HEAD SHA, not for an APPROVED state on existing reviews. The loop is fix -> push -> wait -> fix -> push -> wait, until a review pass lands with no new findings (state becomes COMMENTED with empty body or APPROVED). On this PR (SQR-65) it took 4 review rounds + 9 inline findings before final APPROVED. Source: `observed`.
