# Curated Learnings

This file is the checked-in synthesis layer for durable learnings promoted
out of `~/.gstack/projects/maz-org-squire/learnings.jsonl`.

It is intentionally curated, not a raw dump. Put non-obvious, repeated,
high-signal lessons here when they should survive tool-local runtime state.

If a learning turns into a real architecture decision, write an ADR instead
of treating this file as the permanent decision record.

## Pitfalls

- **ship-pr-body-needs-template-and-linear-close** (pitfall): In Squire, /ship output is not done when the PR merely exists. The PR body must be regenerated to the full ship template, and it must include a Linear closing line like Fixes SQR-84 or Closes SQR-84. Do not accept gh pr create defaults, commit-list bodies, or terse hand-written summaries as the final PR metadata. Files: `docs/agent/shipping.md`, `docs/agent/issue-workflow.md`, `docs/CONTRIBUTING.md`. Source: `observed`.

## Patterns

- **interactive-plan-reviews-need-disk-checkpoints** (pattern): In Codex, long interactive architecture reviews can lose approved decisions to context compaction unless the branch writes checkpoints to a repo file after each answer. For multi-step plan reviews, keep a live checkpoint doc and roll the final approved decisions back into the tech spec once the questioning phase ends. Files: `docs/plans/*.md`, `docs/agent/planning-artifacts.md`. Source: `observed`.
- **browse-stop-after-server-restart** (operational): After restarting the local app during browser QA, gstack browse can keep the old page state and make a fresh fix look broken. Run browse stop or restart the browse daemon before trusting post-restart QA results. Files: `src/server.ts`, `src/web-ui/squire.js`. Source: `observed`.
- **chat-ui-qa-must-include-second-turn-submit** (pattern): When a branch touches shared chat browser code such as HTMX request wiring, SSE handling, or conversation-page form behavior, manual QA must include asking a second question in the same conversation. First-turn submit, seeded transcript rendering, and direct SSE checks are not enough, because follow-up messages use a different request path and can silently regress while the first-turn flow still looks healthy. Files: `docs/agent/qa.md`, `src/web-ui/squire.js`, `src/server.ts`. Source: `observed`.
- **all-branch-changes-go-in-pr** (preference): If anything is changed on the branch, it belongs in the PR. Do not leave branch-local doc or cleanup changes out of the shipping commit set. Files: `docs/agent/shipping.md`, `docs/CONTRIBUTING.md`. Source: `user-stated`.
- **domain-language-beats-storage-jargon** (preference): For operator commands, table names, schema files, and bootstrap/status enums, use the game-domain words users already understand. In Squire, names like `seed:traversal` and `traversal_links` hid the job; `seed:scenario-section-books`, `scenario_book_scenarios`, `section_book_sections`, and `book_references` are clearer because they say exactly what data exists. Files: `package.json`, `src/db/schema/*.ts`, `src/service.ts`. Source: `user-stated`.
- **recent-question-overflow-needs-full-history-projection** (pattern): In the ledger UI, overflow is not just a CSS problem. If older recent questions are truncated in the server-side projection, any drawer or secondary history surface can only reveal the same truncated set. Keep the conversation projection authoritative and let the layout decide which recent questions stay visible versus move behind `More history`. Files: `src/chat/conversation-service.ts`, `src/web-ui/layout.ts`, `src/web-ui/styles.css`. Source: `observed`.
- **multi-book-retrieval-bugs-can-be-contract-drift** (pattern): If `data/pdfs/` already contains the rulebook plus scenario and section books, do not assume a retrieval bug means the indexer skips files. In Squire, `src/index-docs.ts` walks every PDF in that directory; the real regression can be rulebook-only wording, missing source labels, or tests that never exercise scenario/section sources. Files: `src/index-docs.ts`, `src/tools.ts`, `src/vector-store.ts`, `test/index-docs.test.ts`, `test/vector-store.test.ts`. Source: `observed`.
- **scenario-section-research-needs-a-real-two-hop-test** (pattern): A one-hop scenario-to-section check is not enough to prove the scenario/section book research path works. Add at least one real two-hop section chain to tool and agent coverage, plus one live ask that follows repeated explicit read instructions, so the model/tool loop proves it can keep chasing sections instead of stalling after the first hop. Files: `src/agent.ts`, `test/agent.test.ts`, `test/tools.test.ts`. Source: `observed`.
- **scenario-book-pages-must-map-by-printed-header** (pattern): Frosthaven scenario-book PDF filenames and page indexes are not canonical scenario identifiers. A file like `fh-scenario-book-42-61.pdf` can contain continuation pages and scenarios whose printed page header does not match `range.start + pageIndex`. Build scenario ownership from the printed scenario header on the page itself, then parse `Scenario Goals` for the conclusion section and `Section Links` for blue-box section refs. Files: `src/import-scenario-section-books.ts`, `data/pdfs/fh-scenario-book-*.pdf`. Source: `observed`.
- **section-book-extraction-needs-coordinate-aware-pdf-parsing** (pattern): Frosthaven section-book PDFs are not single linear sections. A raw `pdf-parse` text dump smears multiple refs from the same page together and loses which section belongs to which region. For deterministic scenario/section book extraction, use `pdf-parse`'s `pagerender` hook with per-page sentinels and item coordinates, reconstruct lines from positioned text items, split each page by entry row before splitting by entry position, read multi-column prose left-to-right within a section band, tolerate OCR-spaced refs like `3 7. 1`, and parse `Section Links` boxes separately from the main prose so exact section text stays clean while read-next hops still survive. Files: `src/import-scenario-section-books.ts`, `data/pdfs/*.pdf`. Source: `observed`.
- **lower-section-rows-cannot-own-still-active-right-columns** (pattern): On Frosthaven section-book pages, a new section can start in the left column while sections from the row above are still flowing in the middle or right columns. A naive row-wide slice will make the lower section swallow its neighbors. When a row has a single new entry, cap its horizontal ownership using the previous row’s entry x positions, and only let short marker tokens like `60` or `A.` spill across that cap when they share a line with in-band text. Files: `src/import-scenario-section-books.ts`, `data/pdfs/fh-section-book-*.pdf`. Source: `observed`.
- **pdf-prose-reflow-needs-space-before-line-join** (pattern): When flattening extracted PDF prose, strip trailing spaces before newline joins and then replace wrapped `letter + newline + lowercase` boundaries with a real space, not an empty string. Otherwise ordinary wrapped prose silently turns into glued words like `ownbreathing` even when the right section was found. Also treat refs listed inside a `Section Links` box as navigation targets, not as full section-body entries. Files: `src/import-scenario-section-books.ts`, `data/extracted/scenario-section-books.json`. Source: `observed`.
- **codex-shell-needs-explicit-node24-path** (operational): In Codex desktop shells for Squire, repo auto-switch via .nvmrc can miss because the shell starts before landing in the repo cwd, leaving commands on Homebrew Node 25. For Node-sensitive commands, prepend $HOME/.nvm/versions/node/v24.14.0/bin to PATH explicitly instead of assuming nvm auto-use fired. Files: `.nvmrc`, `package.json`. Source: `observed`.
- **vitest-clearallmocks-leaves-once-queues** (operational): In Vitest, vi.clearAllMocks() clears call history but leaves queued mockReturnValueOnce/mockResolvedValueOnce behavior intact. In files with hoisted shared mocks, that can leak one-time return values across tests and show up only under a different execution order such as CI coverage runs. Fix by mockReset() on the affected mocks before re-establishing defaults, rather than relying on clearAllMocks alone. Files: `test/server-api.test.ts`. Source: `observed`.
- **drizzle-kit-beta-breaks-current-drizzle-orm-cli** (operational): In Squire, upgrading drizzle-kit from 0.31.x to the 1.0.0-beta line removes the @esbuild-kit deprecation/audit chain, but the CLI breaks against the current drizzle-orm with ERR_PACKAGE_PATH_NOT_EXPORTED for drizzle-orm/\_relations. Treat drizzle-kit beta adoption as a paired drizzle-kit/drizzle-orm migration, not a safe audit-only bump. Files: `package.json`. Source: `observed`.
- **rebase-needs-test-db-migrate** (operational): After rebasing this repo onto a newer main, db-backed tests can all fail with relation missing errors until npm run db:migrate:test is re-run for the checkout-local test database. Source: `observed`.
